## 1. 技术方案在文档体系中的位置

```
需求文档(requirements/)      → 「要什么」:愿景、范围、功能条目、接口契约(行为级)
        ↓
架构设计(architecture/)      → 「为什么这么选」:技术栈表、关键决策记录、目录结构、待定项
        ↓
技术方案(tech-solution/)     → 「具体怎么实施」:← 本目录
        ├─ README.md        → 总纲(本文):框架、分工、原则、路线、默认决策
        ├─ architecture.md  → 架构详案:运行时拓扑、通信契约、任务执行模型、横切关注
        ├─ database.md      → 数据库设计:ER 模型、表结构、索引、隔离实现、迁移
        ├─ backend.md       → 后端方案:FastAPI 分层、LangGraph 编排、core/llm、加密、测试
        └─ frontend.md      → 前端方案:React 工程、路由、状态、WS/REST 客户端、页面落地
        ↓
openspec/changes/            → 「这次改什么」:把上述方案切成可实施的变更(specs/tasks)
```

**判据**:能问「用哪个库 / 什么算法 / 走什么协议 / 表怎么建 / 模块怎么分」的,都属技术方案(归本目录),不进 `requirements/`;而「为什么选 A 不选 B」的决策记录仍归 `architecture/system-architecture.md`。技术方案引用决策、不复述决策。

**本期范围**(与 [`system-requirements.md`](../requirements/system-requirements.md) §3 一致):仅落地「结构化分析 + 分镜视频生产流水线」子系统,并打好多用户 + BYOK 地基;剧本生成(FR-G)、多角色模拟(FR-S)本期不实现,仅在结构上预留。

## 2. 文档结构与分工

| 子文档 | 回答 | 边界(写什么 / 不写什么) |
|--------|------|--------------------------|
| **architecture.md**(架构详案) | 系统长什么样、各部分怎么协作 | 写:运行时拓扑、进程/部署模型、**前后端通信契约(REST 端点 + WebSocket 帧协议)**、任务执行模型(状态机/并发/取消重试)、横切关注(日志/配置/错误/CORS/可观测性)、目录结构总览。**不写**:具体表结构、具体组件实现。 |
| **database.md**(数据库设计) | 数据怎么存 | 写:ER 模型、表与字段、索引、约束、**用户隔离落地(user_id + 查询过滤)**、加密字段存储形态、JSON 产物字段、字符集与排序、迁移策略。**不写**:ORM 用法细节、业务逻辑。 |
| **backend.md**(后端方案) | 后端怎么搭 | 写:FastAPI 分层(api/core/db/graphs/analysis…)、依赖注入与鉴权、LangGraph 分析图、`core/llm` 单一接缝、任务执行器、富媒体存储抽象、信封加密、配置/错误/测试。**不写**:前端、表 DDL(见 database.md)。 |
| **frontend.md**(前端方案) | 前端怎么搭 | 写:React+TS+Vite 工程结构、路由、状态管理、**WS/REST 客户端封装**、token 分层存储与自动刷新、页面与旅程、多候选交互与流式渲染、表单校验、工具链。**不写**:后端实现、表结构。 |

**四篇不重叠的切分线**:跨端的东西(拓扑、通信契约、任务模型)归 `architecture.md`;落库的东西(ER/表/索引)归 `database.md`;后端进程内部的东西归 `backend.md`;浏览器内部的东西归 `frontend.md`。任一主题只在一个文档为主、其余以「见 §X」指向。

**建议撰写顺序**:`总纲 → architecture.md(契约先行)→ database.md(表是后端 ORM 与前端资源的依据)→ backend.md → frontend.md`。前后端各取所需,但共享同一份契约。

## 3. 系统全景

**一句话**:drama-smith 是前后端分离的 Web 应用,围绕「**剧(Drama)→ 剧集(Episode)**」两级容器,把已有剧本走通一条 **拆解 → 分镜 → 视觉素材 → 逐镜视频 → 合并成片** 的生产流水线;后端用 LangGraph 编排、经 `core/llm` 单一接缝调用用户自带的文本/图片/视频模型,产物与配置落 MySQL、按用户隔离。

### 3.1 运行时拓扑

```
┌────────────────────────┐        REST(JSON)         ┌──────────────────────────────────────────────┐
│  前端(frontend/)       │  ◄───────────────────►   │            后端(backend/)                     │
│  React + TS + Vite      │     WebSocket /ws/tasks    │            FastAPI(Python 3.12+)              │
│  ┌────────────────────┐ │                           │  ┌────────────────────────────────────────┐  │
│  │ 路由/状态/客户端    │ │                           │  │ api 接口层:REST + /ws/tasks            │  │
│  │ 登录/剧库/工作台    │ │                           │  │   依赖注入 · JWT 鉴权 · 用户隔离过滤     │  │
│  │ 分镜台/任务页       │ │                           │  ├────────────────────────────────────────┤  │
│  └────────────────────┘ │                           │  │ graphs:LangGraph 分析图(编排)         │  │
└────────────────────────┘                           │  │ analysis:节点 + 提示工程               │  │
                                                      │  ├────────────────────────────────────────┤  │
                                                      │  │ core:llm 接缝 · config · 加密 · pydantic │  │
                                                      │  │   任务执行器(进程内 async + 持久化)     │  │
                                                      │  ├───────────────┬────────────┬───────────┤  │
                                                      │  │ db:SQLAlchemy │ 富媒体存储  │           │  │
                                                      │  └───────────────┴────────────┘           │  │
                                                      └─────────────┬──────────────┬──────────────┘
                                                                    │              │
                                                          ┌─────────▼──────┐ ┌────▼─────────────┐
                                                          │   MySQL        │ │ 富媒体(图/视频)   │
                                                          │ 用户·配置·任务  │ │ 本期:本地磁盘     │
                                                          │ 产物元数据      │ │ (FileStore 抽象)  │
                                                          └────────────────┘ └──────────┬───────┘
                                                                       LLM 供应商 │ 文本/图片/视频
                                                          ┌──────────────────────────▼──────────┐
                                                          │ core/llm:litellm(文本/图片)+ 自定义适配器(视频等)│
                                                          └─────────────────────────────────────┘
```

### 3.2 核心流水线(数据视角)

```
建剧 → 建剧集(设画幅/风格)→ 剧本输入(可选 AI 优化)→ 预置/引入角色
   → 文本拆解[文本模型] → 角色/情节线/冲突/节奏 + 分镜(3-15s)
   → 分镜编辑(拆/合/排序)
   → 视觉素材[图片模型,多候选,跨镜一致] → 逐镜视频[视频模型,多候选]
   → 合并成片(硬切,无声)→ 导出(mp4 + 台本 + 角色卡)
   ↑ 全程长任务统一持久化 → 任务页(REST 轮询 + WS 实时)双通道回传
```

> 每个标注「模型」的阶段需用户已配置相应模型(BYOK),否则入口禁用并引导补配(见 [`ai-config.md`](../requirements/features/ai-config.md) FR-C1)。

## 5. 核心设计原则(需求 → 原则映射)

| 原则 | 对应需求 | 落地约定 |
|------|----------|----------|
| **多租户强制隔离** | NFR-7、FR-U2 | 所有业务表带 `user_id`;所有查询经「用户过滤」中间件强制带归属;越权访问一律 404/403。**数据库与后端双层把关**(database.md、backend.md)。 |
| **供应商无关单一接缝** | NFR-2 | 文本/图片经 litellm,视频等经 `core/llm` 内自定义适配器;`analysis`/`graphs` **绝不**直接耦合厂商 SDK。切换只改配置、不破子系统。 |
| **凭证安全** | NFR-8、FR-C4 | API Key 信封加密落库,主密钥(MEK)不入库/不入日志;列表脱敏(`sk-…ab12`)。 |
| **认证与会话安全** | NFR-6、FR-U4 | 密码 argon2id;访问令牌 JWT 短时(15min)+ 刷新令牌可吊销;登录失败按账号锁定(5 次/15min)。 |
| **任务可恢复** | FR-A11 | 长任务持久化为记录;状态机 `pending→running→succeeded/failed/canceled`;断线/关页面回来可见已落地产物;单步可独立重做。 |
| **可复现与可维护** | NFR-1/3/4 | uv 锁文件、一条命令起服务/跑测试;ruff+mypy+pytest;清晰模块边界。 |
| **成本可控(BYOK)** | FR-C、§6 成本 | 批量生成前给成本预估与确认;自检不产生真实生成费用。 |
| **本期不做即不建** | system-requirements §7 | 生成/模拟/协作/计费/音频/字幕本期不落地,结构上预留但不实现。 |

## 6. 关键技术决策(承接 architecture + 补齐未定项)

> 已定决策(承接 [`system-architecture.md`](../architecture/system-architecture.md)):① LangGraph 编排、litellm 退居接缝;② 前后端分离、**monorepo**(后端 `backend/` + 前端 `frontend/` 同仓,各独立工具链);③ 长任务**双通道**(REST 轮询基线 + WebSocket `/ws/tasks` 实时,**不用 SSE**);④ 信封加密 API Key;⑤ JWT(HS256)+ 不透明刷新令牌分层存储。
>
> 以下为 architecture §6 中**尚未拍板、但子文档必须有依据**的项,给出**建议默认**(标注「建议」),供用户确认后定稿。

| 决策点 | 建议默认 | 理由 / 边界 |
|--------|----------|-------------|
| **ORM / 迁移** | SQLAlchemy 2.0(async,asyncmy 驱动)+ Alembic | 现有架构已列为候选;async 与 FastAPI 一致;迁移可控、可回溯。 |
| **富媒体存储** | 本期**本地磁盘**(可配根目录)+ `FileStore` 抽象接口;元数据(路径/大小/类型)落 MySQL | 单实例、单用户量级下本地磁盘足够;抽象一层(`save/read/sign-url`)便于后续平滑切 MinIO/S3。视频体积较大,需配额与清理策略。 |
| **任务执行器** | **进程内 asyncio 执行器** + 任务记录持久化 + 重启恢复 | 本期单实例,Celery 过重;任务状态落 MySQL 保证「可关页面回来」;进程重启后把 `running` 置为可重试的中间态由用户重试/重做。多实例时再上消息队列(预留接口)。 |
| **并发上限** | 每用户并发任务上限(建议 3–5)、全局 worker 上限按配置;超限排队 `pending` | BYOK 成本可控 + 保护供应商配额;批量前给预估与确认。 |
| **LangGraph checkpointer** | 本期**不强制**复用 MySQL 做 checkpointer | 任务记录 + 单步重做已满足「回来继续看」;长流程断点续跑留待真正需要时再加。 |
| **JWT 签名密钥 / MEK 注入** | 均经环境变量(`.env`)注入,**不入库不入日志**;生产建议接 KMS | 单实例本地开发 `.env` 足够;预留 KMS 接缝。 |

> 上述建议默认**不在本总纲展开实现**,定稿后分别落入对应子文档(database.md 的迁移与存储元数据、backend.md 的执行器与接缝、architecture.md 的并发与恢复)。
